
# 产品需求文档 (PRD)：栖物志 Item Habitat  
**面向大学生的 AI 轻量化物品管理系统（个人版）**

---

## 1. 背景与目标

### 1.1 背景
大学生及家庭用户常面临“琐碎物品无序堆放、找物困难、闲置不知清理”的痛点。现有收纳 App 多偏向清单记录，缺乏空间可视化与智能化整理流程引导。

### 1.2 目标
打造一个“生活化 + 适合展示”的轻量级整理助手，具备以下核心价值：
- **可视化定位**：将物理储物空间数字映射，实现“按图索骥”。
- **流程化整理**：通过看板式状态流，将整理变为可推进的任务。
- **智能化驱动**：OCR 截图导入与语义搜索，降低录入门槛，生成清理建议。
- **展示性强**：在学院项目展示中体现完整的工程闭环与技术创新。

---

## 2. 目标用户与版本定位

**目标用户**：
- 大学生：宿舍/实验室物品多且杂，需快速归位与借还管理。
- 小型项目团队（未来团队版）：多人共享设备或材料的调配、领用与归还。

**版本策略**：
- 当前为**个人版**，面向独立使用者。采用前后端分离架构，**Python (Flask/FastAPI)** 提供本地 RESTful API 服务，数据存储于轻量级文件数据库 (SQLite)，用户本地启动服务即可使用。
- 团队版将在后续迭代中推出，支持多人通过自建本地服务器协作。当前已在后端接口层与数据模型设计上预留扩展能力，未来仅需升级后端逻辑与数据库。

---

## 3. 核心功能需求

### 3.1 物品与空间管理（核心基座）

**物品属性**：名称、分类（自定义颜色/图标）、照片、数量、存放位置（关联空间）、状态（`pending` / `stored`）、标签、截止时间、备注。团队版预留 `assignee`（负责人）字段，个人版隐藏。

**空间地图**：储物空间（抽屉/柜子/箱子）以网格卡片形式展示，支持上传实景照片与提示语。

**映射关联**：物品必须关联至特定空间，实现“查物即查位”。

**出入库微操作**：在物品详情抽屉/列表中提供 **[取出]** / **[归还]** 按钮，改变物品数量并自动写入操作日志，不改变物品状态。

### 3.2 状态看板与整理流程（两列）

**两列看板**：**待整理** 与 **已归位** 并排展示。
- 卡片在列间**拖拽**直接切换状态（pending ↔ stored），操作同时记录日志。
- 每张卡片除基本信息外，显示**截止时间**和自定义标签。
- 卡片上提供 **“一键归位”按钮**，点击立即将物品移至“已归位”列，无需拖拽。
- 物品进入“已归位”后，其关联的空间卡片浮现“📍最近归位”角标。

### 3.3 截图导入与 OCR

- 支持上传电商/快递截图。
- OCR 提取文本，通过正则/关键词提取商品名称与数量。
- 提供人机协同校对界面，支持修改、合并、指定分类与空间。
- 确认后批量创建物品，初始状态均为“待整理”，并保留图片来源关联。

### 3.4 数据统计与智能提醒

- **图表**：分类数量柱状图、分类占比饼图、近 7/30/90 天使用频率折线图。
- **热度分析**：根据出入库次数区分常用与冷门物品，冷门物品标记提醒。
- **智能推送**：超过 60 天未取出的物品，在首页生成“建议清理”“建议归还”提醒卡片。

### 3.5 智能查询

- 支持自然语言搜索（如“最近没用的无线设备在哪”），通过关键词匹配或轻量 NLP 解析意图，返回物品及其位置。
- 预留后期接入 LLM API 的能力。

## 5. 数据实体定义

以下模型为方案二“回归 PRD，重构为 Item Habitat 整理系统”的正式数据模型，后续实现以本节为准。

**物品 (Item)**：
```typescript
{
  id: string,
  name: string,
  categoryId: string | null,
  imageUrl: string | null,
  quantity: number,
  spaceId: string | null,
  status: "pending" | "stored" | "taken" | "archived",
  tags: string[],
  deadline: string | null,
  notes: string,
  source: "manual" | "receipt_import" | "demo",
  sourceId: string | null,
  lastTakenAt: string | null,
  lastReturnedAt: string | null,
  createdAt: string,
  updatedAt: string
}
```

说明：
- `pending` 表示待整理，`stored` 表示已归位，`taken` 表示已取出使用中，`archived` 表示已清理/归档。
- `quantity` 表示同类物品数量，不将系统主轴设计成传统仓库库存。
- `source/sourceId` 用于追溯手动创建、截图导入或 Demo 数据来源。
- 团队版负责人字段暂不进入个人版主模型，后续可扩展为 `ownerId` / `assigneeId`。

**空间 (Space)**：
```typescript
{
  id: string,
  name: string,
  photoUrl: string | null,
  hint: string,
  parentId: string | null,
  gridPosition: number,
  recentlyOperatedAt: string | null,
  createdAt: string,
  updatedAt: string
}
```

说明：
- `parentId` 用于预留层级空间，例如“衣柜 > 第二层 > 收纳盒 A”。
- `recentlyOperatedAt` 由归位、移动、取出、归还等事件更新，用于地图页“最近归位/最近操作”提示。

**操作日志 (Log)**：
```typescript
{
  id: string,
  itemId: string,
  actionType:
    | "create"
    | "update"
    | "move"
    | "store"
    | "take"
    | "return"
    | "quantity_adjust"
    | "archive"
    | "delete",
  quantityDelta: number | null,
  fromSpaceId: string | null,
  toSpaceId: string | null,
  fromStatus: string | null,
  toStatus: string | null,
  detail: string,
  createdAt: string
}
```

说明：
- 日志是统计、时间线、冷热分析、60 天未使用提醒和空间最近操作角标的统一事件来源。
- 状态变化、空间移动、数量变化都必须写入日志，避免前端各页面各算各的。

**分类 (Category)**：
```typescript
{
  id: string,
  name: string,
  color: string,
  icon: string,
  sortOrder: number,
  createdAt: string,
  updatedAt: string
}
```

**截图导入批次 (ReceiptImport)**：
```typescript
{
  id: string,
  imageUrl: string,
  status: "draft" | "confirmed",
  rawText: string | null,
  draftJson: object,
  createdAt: string,
  confirmedAt: string | null
}
```

说明：
- 截图导入确认后批量创建 `Item`，默认 `status` 为 `pending`。
- `draftJson` 保存校对前后的结构化草稿，便于回看、继续编辑和追溯来源。

**用户 (User，团队版预留)**：
```typescript
{
  id: string,
  name: string,
  avatar: string,
  createdAt: string,
  updatedAt: string
}
```

个人版暂不开放成员管理，数据库可暂缓创建 `users` 表；团队版启动时再引入用户、成员关系和权限字段。

## 6. 核心交互流程

### 6.1 日常整理流
用户打开首页 → 发起 HTTP 请求搜索“蓝牙耳机” → 后端返回匹配物品及空间信息 → 前端展示“快捷取出”结果 → 用户确认取出 → 前端发送 `POST /api/log` 与 `PUT /api/items/{id}` 更新数量 → 后端写入操作日志并返回更新后数据。

### 6.2 网购入库流
进入截图导入页 → 上传淘宝截图 → 前端将图片送至 OCR 处理模块 → 返回识别文本后，通过后端关键词提取接口解析出 3 件商品 → 人机协同校对界面修改、分配合适的空间 → 确认导入 → 前端批量调用 `POST /api/items` → 看板“待整理”列出现 3 张新卡片。

### 6.3 归位闭环流
在看板中，将“蓝牙耳机”卡片拖拽或点击“一键归位” → 前端发送 `PUT /api/items/{id}` 更新状态为 `stored` 并记录日志 → 地图页请求空间数据时，后端返回“书桌抽屉A”的 `recentlyOperatedAt` 已更新 → 前端渲染“📍最近归位”角标 → 统计页调用 `/api/stats` 获取更新后的已归位占比。

## 7. 非功能需求

- **桌面端优先**：采用宽屏布局，看板并排、图表并排，界面清爽高效。保留弹性布局基础，后续可适配移动端。
- **数据存储与技术架构**：  
  - **个人版架构**：采用前后端分离。Python (Flask/FastAPI) 提供本地 RESTful API 服务，前端通过 HTTP 请求进行数据交互。数据持久化于服务端的轻量级文件数据库 (SQLite) 中。  
  - **设计原则**：用户本地运行 Python 后端服务，实现数据物理隔离，保障隐私与离线可用（服务在本地）。  
  - **接口设计**：遵循 RESTful 规范，当前为个人版提供完整的 CRUD 接口。数据访问逻辑完全封装在后端，为未来团队版向多用户、网络化服务器的迁移提供平滑的接口级扩展基础，届时仅需修改后端数据层与服务配置。
- **低门槛操作**：高频操作（取出/归还/归位）不超过 2 步。
- **可演示性**：内置 Demo 数据生成脚本（Python），一键填充示例物品与空间，确保统计图表、提醒功能离线即可展示。
- **性能**：截图导入的 OCR 处理应通过后端 Python 模块在合理时间内完成（可选用 Tesseract OCR 库或其 Python 封装），前端接口响应 <200ms（本地回环网络下）。

## 9. 回归 PRD 重构排期

### 第 1 阶段：数据模型与接口重构（2-3 天）
- 新建或迁移 SQLite 表：`items`、`spaces`、`categories`、`logs`、`receipt_imports`。
- 将现有 `products/storage_locations/transactions` 数据迁移为新的 `items/spaces/logs`。
- 补齐 RESTful API：物品 CRUD、空间 CRUD、分类 CRUD、日志查询、状态流转、取出/归还、截图导入确认。
- 保留现有截图导入链路，但确认后改为创建 `pending` 状态的 `Item`。

交付标准：后端能独立跑通物品、空间、日志、截图导入的完整闭环。

### 第 2 阶段：核心页面回归 PRD（3-4 天）
- 重做看板页：基于 `status` 展示“待整理 / 已归位”，支持拖拽、一键归位、日志写入。
- 重做空间地图页：接入真实 `spaces` 数据，支持照片、提示语、层级空间和空间详情。
- 重做物品详情抽屉：展示完整字段，支持取出、归还、归位、移动空间、标签和备注。
- 增加物品库页：全量列表/网格、筛选、语义搜索入口、快速出入操作。

交付标准：用户能完成“截图导入 → 校对 → 待整理 → 归位 → 查找 → 取出/归还”的主流程。

### 第 3 阶段：统计、提醒与设置（2-3 天）
- 统计页接入分类数量、分类占比、近 7/30/90 天使用趋势。
- 基于 `logs` 计算冷热物品、60 天未使用提醒、建议清理/建议归还卡片。
- 设置页补齐分类管理、后端连接状态、JSON 导入/导出、清空数据、Demo 数据加载。
- 时间线页改为基于 `logs`，支持按日期和操作类型筛选。

交付标准：首页、统计页、时间线页的数据都来自统一日志事件流。

### 第 4 阶段：演示打磨与验收（1-2 天）
- 准备 Demo 数据生成脚本，覆盖宿舍/实验室/项目材料等典型场景。
- 补充关键接口和主流程测试。
- 修正文案，把“库存/货物/出入库”统一替换为“物品/空间/取出/归还/归位”。
- 录制 90 秒演示路径，检查空状态、错误状态和后端未连接状态。

交付标准：项目能稳定演示 PRD 中的核心价值：空间可视化、流程化整理、OCR 导入、智能提醒。
